DNPI 3.0 · Visual implementation plan

One profile-backed verification story — built in three milestones, nine epics

DNPI 3.0 turns Dynamic NPI from a "find every gap" feature into a profile-backed product: consistent statuses across PTApp and the API, alias triangulation against primary sources, a separate pre-go-live file pass, post-go-live monitoring that doesn't auto-dismiss real work, and an internal audit lens for "what changed?" — shipped behind RBAC and pilot flags.

TL;DR — what we are building, in plain English

The four sentences below are the entire program. Everything else in this plan exists to make these four sentences true reliably and at scale.

Pillar F1 One vocabulary, two surfaces

PTApp and the API emit the same status word for the same real-world condition. "Stale" is not "failed". "Not found" is not "we never looked".

Pillar F2 Aliases that triangulate

Name variants are internal hints until the NPI + license + board agree. Then they get promoted to confirmed and help everyone's next lookup.

Pillar F3 Onboarding ≠ monitoring

Pre-go-live file pass is a different journey from live monitoring. Row-level results, no silent drops, reruns supported. Live conflicts show submitted vs primary source, not "fix our profile".

Pillar F4 Internal audit, not customer audit

PT staff get a "what changed?" timeline behind RBAC. A full client-facing audit log is explicitly out of scope for 3.0.

Hard non-scope reminders. Monitor API 2.0 is not a 3.0 gate. There is no client-facing delist API/UI in 3.0. No client-facing full audit. No free-form field pickers that break packaging. PTNetwork-derived demographic mismatch UX stays out.

The story (one page)

DNPI 3.0 is not a single epic — it's a program. The art is shipping the foundation (statuses, origin, RBAC, flags) before anyone leans on it; then shipping the client-visible value (file pass, aliases, conflicts, monitoring); then deciding lifecycle packaging once Product locks "how does a caller ask for initial vs ongoing vs re-cred".

Status model — one vocabulary, two surfaces

The single hardest thing in 3.0 is making sure PTApp and the API say the same word for the same condition. Below is the canonical vocabulary and a mapping note for engineering.

StatusMeaningSurfaces
VerifiedApproved check completed and passedPTApp + API
Not verifiedNo approved check, or check failedPTApp + API + short reason
In progress / PendingCheck in flight within SLAPTApp + API
ConflictSources disagree; client must resolvePTApp + API + accept/reject
Manual / Fee requiredPaid or manual jurisdiction stepPTApp + API + fee context if applicable
Not foundLooked; nothing matched — not "did not look"PTApp + API
On file (unverified)Data present without approved pathPTApp + API; never implies verified
StaleOld timestamp, no hard failurePTApp + API; neutral cue
DeactivatedNPI deactivated federallyPTApp + API; monitoring stops

Engineering note: a translator, not a rewrite

Don't try to repaint every legacy state at once. Stand up a status mapping layer at the serialization boundary that converts internal Monitor/VAP states into the canonical vocabulary. Keep the legacy column in storage; only the wire shape changes.

// Pseudocode — status translator at the API/UI boundary
enum DnpiStatus {
  VERIFIED, NOT_VERIFIED, IN_PROGRESS,
  CONFLICT, MANUAL_FEE_REQUIRED,
  NOT_FOUND, ON_FILE_UNVERIFIED, STALE, DEACTIVATED
}

function toDnpi(internal): DnpiStatus {
  switch (internal.legacyState) {
    case 'AUTO_VERIFIED':        return VERIFIED;
    case 'STALE_NO_FAILURE':      return STALE;
    case 'NPPES_DEACTIVATED':     return DEACTIVATED;
    case 'NO_MATCH_AFTER_LOOKUP': return NOT_FOUND;
    case 'CHECK_NEVER_RAN':
      // CRITICAL: never collapse this into NOT_FOUND
      return IN_PROGRESS;
    case 'SOURCES_DIVERGE':       return CONFLICT;
    case 'FEE_REQUIRED':          return MANUAL_FEE_REQUIRED;
    default:
      // fail-loud: surface a structured warning to ops
      ops.warn('unmapped_legacy_state', internal);
      return ON_FILE_UNVERIFIED;
  }
}
Decision 6 still open. The mapping table from current internal states → canonical DNPI vocabulary is an artifact engineering owes Product before M1 exit. Until then, the default branch above must fail loud in observability rather than silently collapse to a benign label.

Promotion rule (read this carefully)

"The client said it's the same person" is not enough by itself. Triangulation requires NPI + name signal + credential coherence supported by a primary source.

// Promotion guard for Temporary → Confirmed
function canPromote(alias, evidence) {
  return evidence.npi.matches(alias.npi)
      && evidence.licenseNumber.matchesBoard(alias.licenseNumber)
      && evidence.licenseType.matchesBoardTaxonomy(alias.licenseType)
      && nameSignal.coherent(alias.name, evidence.boardName, {
           middle: 'rules',            // per dnpi_credentials.md
           lastSubstring: true,        // at-add only
         })
      && !clientAttestationOnly(evidence);  // hard NO
}
Promotion into the proprietary profile store (PPS) and reuse across clients is Decision 3 — still open. Until that closes, treat promoted aliases as scoped to the originating client and flag them for compliance review before any cross-tenant reuse.

Migration & cutover (Epic 9) — pilot first

The traditional → DNPI conversion is usually a per-client switch, not an org-wide flip. The sequence below is the working playbook.

Pre-cutover checklist

  • Baseline subject count + active alert volume
  • Status mapping table reviewed for this tenant's data
  • Alias confirmations spot-checked
  • RBAC posture verified for tenant's actual users
  • Rollback runbook signed by EM + CS

During cutover — guardrails

  • Alert delta within ±X% of baseline (tune per pilot)
  • No new "fix our profile" wording on any client surface
  • Provenance shifts produce no duplicate alerts
  • Termed / deactivated subjects excluded from re-cycles

If we have to roll back

  • Flip the tenant flag → behavior reverts to legacy paths
  • Capture a postmortem with delta tables + alert diff
  • Update Decision 7 evidence with what we learned
